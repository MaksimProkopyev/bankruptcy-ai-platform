# План тестирования RAG v2

## Цель
Убедиться, что новые эндпоинты `/knowledge/*` работают корректно, интегрированы с БД и компонентами RAG v2.

## Область тестирования
1. **Эндпоинты API** (8 штук):
   - `POST /knowledge/search`
   - `POST /knowledge/ask`
   - `GET /knowledge/sources`
   - `POST /knowledge/sources`
   - `POST /knowledge/sources/{source_id}/ingest`
   - `DELETE /knowledge/sources/{source_id}`
   - `POST /knowledge/feedback`
   - `GET /knowledge/stats`

2. **Интеграция с БД**:
   - Подключение к PostgreSQL через async SQLAlchemy.
   - Корректность миграции `003_rag_v2`.
   - CRUD операции с таблицами `knowledge_sources`, `knowledge_chunks`, `rag_queries`, `rag_feedback`.

3. **Компоненты RAG v2**:
   - Ingestion pipeline (парсеры, чанкирование, эмбеддинг, индексация).
   - Retrieval (гибридный поиск, реранкинг).
   - Generation (построение контекста, генерация ответов).

## Типы тестов
### 1. Unit-тесты
- Мокирование внешних зависимостей (OpenAI API, Claude API).
- Тестирование отдельных классов: `QueryProcessor`, `LegalChunker`, `EmbeddingService`, `HybridRetriever` и т.д.
- Проверка бизнес-логики.

### 2. Интеграционные тесты
- Тестирование эндпоинтов с тестовой БД (использование `pytest-asyncio`, `httpx`).
- Проверка полного пайплайна ingestion → search → answer.

### 3. E2E-тесты
- Симуляция пользовательского сценария: загрузка источника → индексация → поиск → генерация ответа → отправка фидбека.

## Инструменты
- `pytest` + `pytest-asyncio`
- `httpx` для асинхронных HTTP-запросов к AI Core.
- `docker-compose` для поднятия тестового окружения (PostgreSQL, Redis).
- `factory_boy` или `pytest-factoryboy` для создания тестовых данных.

## Этапы тестирования
### Этап 1: Подготовка инфраструктуры
1. Поднять тестовые базы данных (используя `docker-compose.test.yml`).
2. Применить миграции (включая `003_rag_v2`).
3. Запустить AI Core на тестовом порту (например, 8002).

### Этап 2: Unit-тесты компонентов
Написать тесты для каждого модуля в `ai-core/rag/`.

### Этап 3: Интеграционные тесты эндпоинтов
Использовать `AsyncClient` FastAPI для тестирования маршрутов.

### Этап 4: E2E-сценарий
Скрипт, который:
1. Создаёт источник (`POST /knowledge/sources`).
2. Запускает индексацию (`POST /knowledge/sources/{source_id}/ingest`).
3. Выполняет поиск (`POST /knowledge/search`).
4. Задаёт вопрос (`POST /knowledge/ask`).
5. Отправляет фидбек (`POST /knowledge/feedback`).
6. Проверяет статистику (`GET /knowledge/stats`).

## Критерии успеха
- Все unit-тесты проходят.
- Интеграционные тесты проходят с изолированной БД.
- E2E-сценарий выполняется без ошибок.
- Время ответа эндпоинтов не превышает 2 секунд (за исключением индексации).
- Покрытие кода > 80% для модулей RAG v2.

## Документация
- Результаты тестирования записывать в `TEST_RESULTS.md`.
- Известные проблемы и ограничения — в `KNOWN_ISSUES.md`.

## Следующие шаги
После успешного тестирования можно считать RAG v2 готовым к интеграции с бэкендом и фронтендом.
