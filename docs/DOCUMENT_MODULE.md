# Модуль документов

## Цель
Обеспечить полный цикл работы с документами в системе банкротства:
- Загрузка и хранение
- OCR и извлечение структурированных данных
- Валидация и чеклист
- Генерация документов
- Интеграция с AI Core

## Текущее состояние

### Существующие компоненты
1. **API документов** (`/api/v1/documents`) – загрузка, скачивание, список, валидация.
2. **Модель Document** – поля для OCR текста, extracted_data, confidence.
3. **Агент OCR** (`ai-core/agents/ocr_extraction.py`) – классификация и извлечение данных из текста.
4. **Сервис чеклиста** (`document_checklist.py`) – определение необходимых документов.
5. **Файловое хранилище** (S3/MinIO) – загрузка через `file_storage.py`.
6. **Очередь задач** (`AITask`) – задачи на обработку документов.

### Недостающие компоненты
1. **OCR движок** – извлечение текста из изображений/PDF.
2. **Фоновая обработка** – воркер, который берёт задачи из `AITask` и выполняет OCR + извлечение.
3. **Интеграция AI Core** – эндпоинт `/ocr` должен быть реализован.
4. **Генерация документов** – шаблоны для заявлений, реестров и т.д.
5. **RAG поиск** – поиск по законодательной базе.
6. **Валидация по чеклисту** – автоматическое сопоставление загруженных документов с чеклистом.

## Архитектура

### 1. OCR Pipeline
```
Загруженный файл → Хранилище → Создание AITask (ocr) → Воркер AI Core
```
Воркер AI Core:
- Загружает файл из хранилища.
- Запускает OCR движок (Surya/Tesseract) для извлечения текста.
- Передаёт текст агенту `OCRAgent` для классификации и извлечения данных.
- Обновляет документ: `ocr_text`, `extracted_data`, `ai_confidence`, `status`.
- Создаёт событие `document_processed`.

### 2. Фоновая обработка
Используем Redis + RQ или Celery. В текущей архитектуре уже есть `AITask` – можно расширить воркером, который опрашивает задачи.

### 3. Генерация документов
Шаблоны в формате Jinja2 + данные из кейса. Генерация PDF через WeasyPrint или docx через python-docx.

### 4. Чеклист и прогресс
Сервис `document_checklist` уже реализован. Нужно добавить эндпоинт для получения чеклиста по кейсу и автоматическое обновление прогресса при загрузке документов.

### 5. RAG для законодательства
Индексирование ключевых статей ФЗ-127, судебной практики. Поиск через векторную БД (pgvector). Эндпоинты `/rag/search` и `/rag/answer` в AI Core.

## План реализации

### Этап 1: OCR Pipeline
1. Реализовать OCR движок в `ai-core/ocr/engine.py` (обёртка над Surya/Tesseract).
2. Реализовать воркер `ai-core/worker.py`, который обрабатывает задачи типа `ocr`.
3. Реализовать эндпоинт `/ocr` в AI Core, который вызывает OCR движок и агента извлечения.
4. Интегрировать с API документов: после загрузки создавать `AITask` и запускать воркер.

### Этап 2: Генерация документов
1. Создать шаблоны в `ai-core/templates/`.
2. Реализовать `DocumentGenerator` в `ai-core/agents/document_generator.py`.
3. Добавить эндпоинт `/generate-document` в AI Core.
4. Интегрировать с API: кнопка "Сгенерировать заявление" в интерфейсе.

### Этап 3: Чеклист и валидация
1. Добавить эндпоинт `GET /cases/{case_id}/document-checklist` с прогрессом.
2. Автоматически обновлять прогресс при загрузке/удалении документа.
3. Добавить уведомления о недостающих документах.

### Этап 4: RAG
1. Заполнить базу знаний статьями ФЗ-127, типовыми решениями.
2. Реализовать индексацию через `rag/pipeline.py`.
3. Реализовать поиск и ответы через LLM.

## Технологии
- **OCR**: Surya (лучше для русского) или Tesseract с русским языком.
- **Очередь задач**: Redis RQ (просто) или Celery (мощнее).
- **Генерация PDF**: WeasyPrint (HTML → PDF) или ReportLab.
- **Векторный поиск**: pgvector (уже в PostgreSQL) или Qdrant.
