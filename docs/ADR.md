# Architecture Decision Records (ADR)

## ADR-001: Монорепо

**Решение:** Единый репозиторий для всех компонентов (backend, ai-core, frontend, mobile, infra).

**Причины:**
- Упрощает CI/CD на старте
- Единые версии зависимостей
- Атомарные коммиты затрагивающие несколько компонентов
- Переход на полирепо позже — тривиален

---

## ADR-002: Кастомная CRM вместо Битрикс24/AmoCRM

**Решение:** Полностью кастомная CRM с нуля на FastAPI + PostgreSQL.

**Причины:**
- Полный контроль над моделью данных (карточка дела — не стандартная CRM-сделка)
- Глубокая интеграция с AI-ядром (AI пишет в те же таблицы, не через API коннекторы)
- Специфичная воронка банкротства (22 статуса, а не 5-7 стандартных)
- Отсутствие vendor lock-in
- Возможность лицензирования как SaaS в будущем

**Риски:** Больше времени на разработку базовой функциональности.
**Митигация:** Фокус на MVP — только необходимые фичи на каждом спринте.

## ADR-003: AI Core как отдельный сервис

**Решение:** AI-ядро — отдельный FastAPI-сервис (порт 8001), общающийся с бэкендом через HTTP API.

**Причины:**
- Независимое масштабирование (AI-задачи ресурсоёмкие)
- Изоляция зависимостей (OCR, ML-библиотеки отдельно от бизнес-логики)
- Возможность независимого деплоя при обновлении моделей/промптов
- Проще тестировать агентов изолированно

**Коммуникация:** Синхронные вызовы для быстрых задач (квалификация), асинхронная очередь (Redis) для тяжёлых (OCR, генерация документов).

## ADR-004: PostgreSQL + pgvector (без отдельной векторной БД)

**Решение:** Использовать pgvector для RAG-эмбеддингов вместо отдельного Qdrant/Pinecone.

**Причины:**
- Одна БД вместо двух — проще операционно
- pgvector достаточен для <1M документов (наш масштаб на 2 года)
- Транзакционная целостность с основными данными
- IVFFlat индекс обеспечивает приемлемую скорость

**Когда пересмотреть:** При >500K чанков или требовании <50ms latency на поиск.

## ADR-005: Claude API как основной LLM

**Решение:** Anthropic Claude как основной LLM, OpenAI GPT-4o как fallback.

**Причины:**
- Лучшее следование инструкциям в системных промптах
- Длинный контекст (200K токенов) для анализа документов
- Хорошая работа с русским языком
- Structured output (JSON mode)

**Fallback-логика:** Если Claude API недоступен или timeout >10s → переключение на GPT-4o. Логирование каждого fallback.

## ADR-006: Human-in-the-loop для всех юридических документов

**Решение:** Ни один документ, сгенерированный AI, не отправляется в суд без проверки юристом.

**Причины:**
- Юридическая ответственность на фирме
- AI может галлюцинировать (особенно ссылки на статьи закона)
- Регуляторные требования
- Доверие клиентов

**Реализация:** Статусы документа: draft → ai_generated → lawyer_review → approved → submitted.

## ADR-007: Multi-tenancy через row-level isolation

**Решение:** Все данные в одной БД, изоляция через client_id/case_id + RBAC на уровне API.

**Причины:**
- Проще на старте
- Одна схема миграций
- Достаточно при <1000 клиентов одновременно

**Когда пересмотреть:** При появлении партнёрских фирм (SaaS-режим) — переход на schema-per-tenant или отдельные БД.

## ADR-008: Хранение данных на территории РФ (152-ФЗ)

**Решение:** Вся инфраструктура — Yandex Cloud (зона ru-central1).

**Причины:**
- ФЗ-152 требует хранения ПД на территории РФ
- Клиентские данные содержат паспортные данные, ИНН, СНИЛС
- Yandex Cloud предоставляет managed PostgreSQL, S3, Redis

**Исключение:** LLM API (Claude, GPT) работают за рубежом. Передаваемые данные анонимизируются: только финансовые суммы, типы кредиторов, статусы — без ФИО, паспортов, ИНН.
